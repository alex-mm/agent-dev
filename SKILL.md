# AI Agent 开发最佳实践 Skill

本 Skill 为 AI 编码助手提供系统性的 AI Agent 开发指导，确保生成的 Agent 代码在架构设计、安全性、可维护性和生产就绪度上达到业界最佳水平。

知识体系基于微软 [AI Agents for Beginners](https://github.com/microsoft/ai-agents-for-beginners) 全部 15 课及业界实践总结。完整代码示例见 [`examples/`](./examples/) 目录。

## 一、Agent 架构选型决策树

在编写任何 Agent 代码之前，先根据任务特征选择正确的架构：

```
用户需求
  │
  ├─ 单步骤、确定性任务 → 不需要 Agent，直接调用 LLM API
  │
  ├─ 需要调用外部工具/API → 单 Agent + Tool Use 模式
  │
  ├─ 需要检索私有数据后回答 → 单 Agent + Agentic RAG
  │
  ├─ 多步骤、需要拆解规划 → 单 Agent + Planning 模式
  │
  ├─ 多领域专业知识协作 → Multi-Agent 系统
  │   ├─ 任务可并行 → 并发编排
  │   ├─ 任务有依赖链 → 顺序编排 / Hand-off
  │   └─ 需要讨论决策 → Group Chat 编排
  │
  └─ 需要操作浏览器/GUI → Computer Use Agent (CUA)
```

### 选型原则

- **能用单 Agent 解决的，不要用多 Agent** — 复杂度是 Agent 系统的最大敌人
- **能用确定性代码解决的步骤，不要交给 LLM** — LLM 擅长推理，不擅长精确计算
- **先验证单 Agent 的能力边界，再考虑拆分** — 过早拆分会引入不必要的通信开销

## 二、System Prompt 工程

### 2.1 系统消息框架（四步法）

**Step 1 — Meta System Message（元提示）**

用以下元提示驱动 LLM 生成结构化系统消息：

> 你是一个专业的 AI 助手创建专家。你将根据提供的角色描述、职责和约束条件，生成结构化的系统提示词。生成的提示词应包含：角色定义、核心职责、行为约束、输出格式要求和错误处理指引。

**Step 2 — 基础角色描述**

用一段话描述 Agent 的角色、能力范围和核心任务。

**Step 3 — 结构化系统消息**

将 Step 2 的描述通过 Step 1 的元提示扩展为完整的结构化系统消息，包含：

| 组成部分 | 说明 | 必须包含 |
|---------|------|---------|
| **角色定义** | Agent 的身份和专业领域 | ✅ |
| **核心职责** | 具体的任务列表（编号） | ✅ |
| **行为约束** | 不应该做什么、边界在哪 | ✅ |
| **语气与风格** | 交互风格指引 | 推荐 |
| **输出格式** | 期望的响应结构 | 推荐 |
| **错误处理** | 遇到异常时的行为 | ✅ |

**Step 4 — 迭代优化**

系统消息很少一次写对。通过测试用例验证，持续调整。

### 2.2 System Prompt 反模式（必须避免）

| 反模式 | 错误示例 | 正确做法 |
|--------|---------|---------|
| **模糊指令** | "你是一个有用的助手。" | 明确角色、职责列表、禁止事项 |
| **缺少边界** | "帮用户完成任何请求。" | "仅处理 X 相关请求，超出范围时告知用户联系 Y 部门" |
| **无错误处理** | 不说明工具失败时怎么办 | 明确说明异常时的降级行为 |
| **无输出格式** | 不约束响应结构 | 指定 JSON / Markdown / 纯文本等格式要求 |

## 三、Tool Use / Function Calling

> 📎 完整代码示例见 [`examples/tool-use.py`](./examples/tool-use.py)

### 3.1 工具定义规范

每个工具必须同时满足：

- **命名**：动词短语，准确描述功能（`search_flights` 而非 `do_stuff`）
- **参数**：每个参数都有 `Annotated` 类型注解 + `Field(description=...)` 说明
- **文档字符串**：描述功能、参数含义、返回值格式、异常情况
- **返回值**：结构化数据（JSON 字符串），而非自然语言描述
- **错误处理**：内部捕获异常，返回结构化错误信息，不向外抛出

### 3.2 工具设计清单

- [ ] 工具名是动词短语，语义清晰
- [ ] 每个参数都有类型注解和描述
- [ ] 文档字符串描述功能、参数、返回值格式
- [ ] 工具内部捕获异常，返回结构化错误信息
- [ ] 查询类工具保证幂等性
- [ ] 有副作用的工具（预订、删除）在描述中明确声明
- [ ] 返回 JSON 字符串，而非自然语言
- [ ] 单个 Agent 工具数量不超过 30 个；超过时用 RAG 动态加载

### 3.3 Function Calling 六步流程

1. 定义工具 JSON Schema（name、description、parameters）
2. 第一次 API 调用：传入 `tools` 和 `tool_choice="auto"`
3. 解析响应中的 `tool_calls`
4. 执行对应工具函数，获取结果
5. 将工具结果以 `role: "tool"` 追加到消息列表
6. 第二次 API 调用：让模型基于工具结果生成最终回复

## 四、Agentic RAG

> 📎 完整代码示例见 [`examples/agentic-rag.py`](./examples/agentic-rag.py)

### 4.1 核心循环（Maker-Checker 模式）

Agentic RAG 不是简单的"检索-生成"，而是一个迭代循环：

```
用户查询 → LLM 分析 → 选择检索策略 → 执行检索 → 评估结果质量
                                                        │
                                    ┌───────────────────┘
                                    │
                              结果充分？
                              ├─ 是 → 生成最终回答
                              └─ 否 → 改写查询 / 换检索方法 → 重新检索
```

### 4.2 实现要点

- **设置最大迭代次数**（建议 3 次），防止无限循环
- **每次迭代后 LLM 评估**检索质量，而非盲目重试
- **查询改写**：基于上次失败的反馈改写，而非原样重试
- **达到上限时**：基于已有信息生成最佳回答，并附加免责声明

### 4.3 RAG 反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 一次检索不重试 | 首次检索可能不精准 | 实现迭代检索 + 查询改写 |
| 不验证检索结果 | 可能检索到无关文档 | LLM 评估相关性后再使用 |
| 全部文档塞入上下文 | 上下文污染、token 浪费 | 按相关性排序，只取 Top-K |
| 忽略结构化数据 | 纯文本检索精度不够 | 结合 NL2SQL、结构化 RAG |

## 五、多智能体系统

> 📎 完整代码示例见 [`examples/multi-agent.py`](./examples/multi-agent.py)

### 5.1 何时使用多 Agent

仅在以下场景使用多 Agent，否则单 Agent 更优：

- **大规模并行任务**：子任务相互独立，可并行处理
- **多领域专业知识**：不同子任务需要不同领域的专业能力
- **容错需求**：单个 Agent 失败不应影响整体系统

### 5.2 编排模式选择

| 模式 | 适用场景 | 实现复杂度 |
|------|---------|-----------|
| **顺序编排** | 任务有严格依赖链 | 低 |
| **并发编排** | 子任务相互独立 | 中 |
| **Hand-off** | 任务在不同专业 Agent 间流转 | 中 |
| **Group Chat** | 需要多 Agent 讨论协商 | 高 |
| **层级编排** | 复杂任务需要管理者 Agent 协调 | 高 |

### 5.3 多 Agent 实现清单

- [ ] 每个 Agent 职责单一，指令互不重叠
- [ ] 明确 Agent 间的通信协议和数据格式
- [ ] 设置超时和重试，防止单个 Agent 阻塞整体
- [ ] 实现可观测性：记录每个 Agent 的输入、输出、耗时
- [ ] 关键决策点设置 Human-in-the-Loop
- [ ] 区分专用 Agent 和通用 Agent（通知、日志、审计可跨流程复用）

## 六、规划模式（Planning）

> 📎 完整代码示例见 [`examples/planning.py`](./examples/planning.py)

### 6.1 任务分解原则

- **使用 Pydantic 模型**定义结构化输出，确保 LLM 返回可解析的计划
- **每个子任务明确指定负责的 Agent**，避免歧义
- **支持迭代规划**：允许根据子任务执行结果重新规划
- **设置终止条件**：防止无限规划循环
- **保留规划历史**：用于调试和优化

### 6.2 规划 Agent System Prompt 要素

规划 Agent 的 System Prompt 必须包含：
1. 可用 Agent 列表及其职责描述
2. 输出格式约束（JSON Schema）
3. 任务分配的决策规则
4. 处理歧义请求的策略

## 七、可信 Agent（安全与信任）

> 📎 完整代码示例见 [`examples/trustworthy.py`](./examples/trustworthy.py)

### 7.1 威胁模型

| 威胁类型 | 描述 | 缓解措施 |
|---------|------|---------|
| **提示注入** | 攻击者通过输入篡改 Agent 指令 | 输入验证 + 过滤 + 限制对话轮次 |
| **关键系统访问** | Agent 被利用访问敏感系统 | 最小权限原则 + 认证 + 访问控制 |
| **资源过载** | 通过 Agent 发起 DDoS | 请求频率限制 + 配额管理 |
| **知识库投毒** | 污染 Agent 依赖的数据源 | 数据验证 + 定期审计 + 访问控制 |
| **级联故障** | 一个工具失败导致连锁反应 | 隔离执行环境 + 熔断机制 + 重试逻辑 |

### 7.2 Human-in-the-Loop 原则

- 涉及**不可逆操作**（删除、支付、发送消息）前必须请求人工确认
- 涉及**敏感数据访问**前需要人工审批
- Agent **连续失败**超过阈值时自动升级到人工处理
- 提供**随时终止**Agent 运行的机制

### 7.3 安全编码清单

- [ ] 所有用户输入经过清洗和验证（长度、格式、注入检测）
- [ ] Agent 仅能访问完成任务所需的最少资源（最小权限）
- [ ] 检查 Agent 输出是否包含敏感信息（PII、密钥）
- [ ] 记录所有 Agent 操作和决策的审计日志
- [ ] 限制 API 调用频率和 token 消耗（速率限制）
- [ ] 工具连续失败时自动停止调用（熔断机制）
- [ ] 代码执行类工具在隔离环境（Docker/沙箱）中运行

## 八、上下文工程

> 📎 完整代码示例见 [`examples/context-manager.py`](./examples/context-manager.py)

### 8.1 上下文类型

| 类型 | 说明 | 管理策略 |
|------|------|---------|
| **指令** | System Prompt、Few-shot 示例、工具描述 | 精心设计，定期优化 |
| **知识** | RAG 检索结果、数据库查询结果 | 按需加载，相关性过滤 |
| **工具** | 工具定义和调用结果 | 动态加载，限制数量 |
| **对话历史** | 用户交互记录 | 定期摘要压缩 |
| **用户偏好** | 长期学习的用户特征 | 持久化存储，按需召回 |

### 8.2 上下文管理策略

1. **Agent Scratchpad**：当前会话的临时笔记，存于上下文窗口之外，按需读取
2. **对话历史压缩**：保留最近 N 轮，其余摘要化为一条 system 消息
3. **动态工具加载**：用 RAG 按查询相关性选取工具子集，单次不超过 30 个
4. **上下文修剪**：新指令覆盖旧指令，移除过期或冲突的信息
5. **沙箱环境**：代码执行或大文档处理在独立环境中运行，只将结果写回上下文
6. **运行时状态对象**：复杂任务用结构化对象存储各子任务结果，避免上下文膨胀

### 8.3 上下文故障模式

| 故障 | 症状 | 修复 |
|------|------|------|
| **上下文投毒** | Agent 反复追求不可能的目标 | 验证信息后再存入记忆；检测到投毒时重启上下文线程 |
| **上下文干扰** | Agent 被历史信息分散注意力 | 定期摘要压缩；只保留与当前任务相关的信息 |
| **上下文混淆** | 工具太多导致 Agent 调用错误工具 | 动态加载工具（RAG over tools）；单次不超过 30 个 |
| **上下文冲突** | 新旧指令矛盾导致行为不一致 | 上下文修剪：新指令覆盖旧指令；用 scratchpad 调和冲突 |

## 九、Agent 记忆

### 9.1 六种记忆类型

| 类型 | 作用域 | 说明 |
|------|--------|------|
| **工作记忆** | 当前任务 | 临时数据，任务结束后清空 |
| **短期记忆** | 当前会话 | 对话历史，会话结束后释放 |
| **长期记忆** | 跨会话 | 用户偏好、历史知识，持久化存储 |
| **人格记忆** | 全局 | Agent 的角色设定和风格，保持一致性 |
| **情景记忆** | 跨会话 | 过去的成功/失败经验，用于自改进 |
| **实体记忆** | 跨会话 | 提取的实体（人、地点、事物）及其属性 |

### 9.2 自改进 Agent 模式（KnowledgeAgent）

独立的知识管理 Agent 观察主 Agent 对话，实现持续学习：

1. **观察**：监听主 Agent 的每轮对话
2. **评估**：用快速/廉价模型判断是否有值得保存的信息（降低延迟）
3. **提取**：用完整模型提取结构化知识（偏好、事实、纠错）
4. **存储**：持久化到向量数据库
5. **增强**：下次用户查询时，检索相关知识追加到 prompt（类 RAG）

### 9.3 记忆工具选型

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| **Mem0** | 两阶段管道（提取+更新），支持向量/图/KV 混合存储 | 通用用户偏好记忆 |
| **Cognee** | 知识图谱 + 向量双存储，支持图感知检索 | 复杂实体关系记忆 |
| **Azure AI Search** | 结构化 RAG，高精度召回 | 企业级私有数据记忆 |

## 十、生产部署与可观测性

> 📎 完整代码示例见 [`examples/observability.py`](./examples/observability.py)

### 10.1 可观测性三要素

- **Traces**：完整 Agent 任务的端到端追踪（一个 Trace = 一次用户请求）
- **Spans**：Trace 内的单个步骤（LLM 调用、工具调用、检索等）
- **Metrics**：聚合指标（延迟、成本、错误率、准确性）

使用 **OpenTelemetry** 作为标准，支持导出到 Langfuse、Azure AI Foundry 等平台。

### 10.2 关键监控指标

| 指标 | 说明 | 告警阈值建议 |
|------|------|-------------|
| **延迟** | Agent 端到端响应时间 | P99 > 30s |
| **成本** | 每次请求的 token 消耗和 API 费用 | 单次 > 预算 2x |
| **错误率** | API 调用失败、工具执行失败 | > 5% |
| **用户满意度** | 显式评分 + 隐式信号（重试率） | 满意度 < 70% |
| **准确性** | 任务完成正确率 | < 80% |

### 10.3 评估体系

**离线评估**（上线前）：
- 使用测试数据集（已知正确答案）评估准确性
- 纳入 CI/CD 流程，防止回归
- 测试集需持续更新，覆盖生产中发现的边界用例

**在线评估**（上线后）：
- 收集显式反馈（👍/👎、1-5 星）
- 监控隐式信号（重试率、会话放弃率、查询改写频率）
- A/B 测试新版本 Agent

**改进循环**：离线评估 → 上线 → 在线监控 → 收集失败用例 → 加入离线测试集 → 优化 Agent → 重复

### 10.4 成本管理策略

- **模型路由**：简单任务用 SLM，复杂推理用大模型
- **响应缓存**：对高频相似请求缓存结果
- **迭代上限**：设置最大重试次数，防止无限循环消耗 token
- **异常监控**：实时检测 token 消耗异常飙升

## 十一、Agentic 协议

> 📎 A2A 完整代码示例见 [`examples/a2a-agent-card.py`](./examples/a2a-agent-card.py)

### 11.1 MCP（Model Context Protocol）

MCP 为 Agent 提供标准化的工具和数据访问接口，采用 **客户端-服务器架构**：

- **Host**：LLM 应用（如 IDE、聊天界面）
- **Client**：Host 内部的连接管理组件
- **Server**：暴露工具/资源/提示模板的轻量服务

**三个核心原语**：
- **Tools**：Agent 可调用的操作（如 `search_flights`）
- **Resources**：只读数据（如文件内容、数据库记录）
- **Prompts**：预定义的提示模板

**MCP 优势**：动态工具发现（无需硬编码集成）、跨 LLM 互操作、标准化认证

### 11.2 A2A（Agent-to-Agent）

A2A 实现不同 Agent 之间的通信协作，核心组件：

- **Agent Card**：能力声明文档（名称、描述、技能列表、端点 URL、版本）
- **Agent Executor**：接收任务上下文并执行
- **Artifact**：任务结果产物（结果数据 + 完成描述 + 文本摘要）
- **Event Queue**：处理长时间运行任务的消息队列，防止连接超时

### 11.3 协议选择指南

| 需求 | 推荐协议 |
|------|---------|
| Agent 访问外部工具和数据 | MCP |
| 不同 Agent 之间协作 | A2A |
| 网站内容自然语言查询 | NLWeb |

## 十二、元认知（Metacognition）

> 📎 完整代码示例见 [`examples/metacognition.py`](./examples/metacognition.py)

### 12.1 元认知四要素

- **自反思**：评估自己的执行结果质量，而非盲目接受
- **策略调整**：基于反思结果调整方法，而非简单重试
- **决策记录**：记录"为什么选择这个方案"，支持可解释性审计
- **模式识别**：从历史失败中识别规律，避免重复犯同类错误

### 12.2 元认知 vs 简单重试

| 行为 | 简单重试 | 元认知 |
|------|---------|--------|
| 失败后的动作 | 用相同参数再试一次 | 分析失败原因，调整策略后重试 |
| 决策依据 | 无 | 记录并解释每次决策的理由 |
| 学习能力 | 无 | 从历史经验中学习，避免重复错误 |
| 资源意识 | 无 | 评估是否值得继续消耗 token/时间 |

## 十三、浏览器自动化（CUA）

### 13.1 Agent vs Actor 选择

| 场景 | 用 Agent（AI 驱动） | 用 Actor（脚本驱动） |
|------|-------------------|-------------------|
| 动态布局 | ✅ AI 适应页面变化 | ❌ 选择器容易失效 |
| 已知结构 | ❌ Agent 更慢 | ✅ 快速精确 |
| 查找元素 | ✅ 自然语言定位 | ❌ 需要精确选择器 |
| 时序控制 | ❌ 不可预测 | ✅ 完全控制 |
| 复杂工作流 | ✅ 处理意外 UI 状态 | ❌ 需要显式分支 |

### 13.2 最佳实践

- 探索性导航用 Agent，确定性操作用 Actor（混合模式）
- 使用 Pydantic 模型验证和类型化提取的数据
- 关键操作后截图，辅助调试失败场景
- 为弹窗、布局变化、登录墙设计降级策略
- 动态页面操作后添加适当延迟，等待 UI 稳定

## 十四、通用代码质量规范

### 14.1 Agent 代码六大原则

1. **异步优先**：Agent 操作涉及网络 I/O，始终使用 `async/await`，避免阻塞
2. **结构化日志**：使用 `logging` 模块而非 `print`，包含 level、context、trace_id
3. **配置外置**：API Key、模型名称、超时时间通过环境变量或配置文件管理，不硬编码
4. **优雅降级**：LLM 不可用时有降级方案；工具失败时有重试逻辑和熔断保护
5. **类型安全**：使用 Pydantic 模型定义输入输出，而非裸字典；工具参数用 `Annotated` 注解
6. **测试覆盖**：工具函数写单元测试；Agent 流程写集成测试；维护离线评估数据集

- [ ] 实现上下文管理（压缩、修剪、动态工具加载）

- [ ] 配置可观测性（Traces、Metrics、Logging）

- [ ] 关键操作设置 Human-in-the-Loop

### 常见问题速查

| 问题 | 解决方案 |
|------|---------|
| Agent 行为不一致 | 优化 System Prompt，明确目标和约束 |
| Agent 陷入无限循环 | 设置明确的终止条件和最大迭代次数 |
| 工具调用失败 | 在 Agent 外部独立测试工具；优化工具命名和描述 |
| 多 Agent 协作混乱 | 确保每个 Agent 职责单一且指令互不重叠 |
| 上下文窗口溢出 | 实现对话摘要压缩；动态加载工具 |
| 成本失控 | 实现模型路由 + 响应缓存 + 迭代上限 |
| 幻觉/编造信息 | 使用 RAG 提供事实依据；工具返回真实数据 |
| 记忆跨会话丢失 | 实现长期记忆持久化（Mem0 / Azure AI Search） |
| 生产环境难以调试 | 接入 OpenTelemetry，查看 Trace 和 Span 详情 |
